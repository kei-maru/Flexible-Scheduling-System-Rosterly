# Rosterly 図解資料メモ

## 1. 目的

本資料は、`Rosterly` の紹介資料向けに、以下 2 点をわかりやすく整理するためのものです。

- データベースER図
- 全体システム構成図

今回の方針は、開発者向けの完全図ではなく、営業資料・提案資料として見やすいことを優先しています。

## 2. 対外説明の基本表現

資料では次の表現に統一する想定です。

- `Rosterly`：マルチテナント型の予約・シフト管理SaaS基盤
- `外部サービス連携`：外部フロントや外部業務システムからSSO経由で接続される利用チャネル
- `公開予約ページ`：Rosterly をログインなしで直接利用できる予約チャネル
- `テナント`：店舗・運営主体
- `ユーザー`：権限を持つログインアカウント
- `スタッフ`：予約対象となる業務リソース
- `予約`：顧客の予約注文
- `サービス`：店舗ごとの提供メニュー

短い説明文：

`Rosterly` が共通業務基盤を担い、外部サービスとはSSO/APIで連携し、直接予約も受け付ける構成です。

## 3. ER図の見せ方

ER図は、あえて以下の 4 ブロックだけに絞ると読みやすくなります。

1. 店舗と権限
2. スタッフ管理
3. シフトとサービス
4. 予約と通知

今回の簡略版ER図に含める主なエンティティ：

- `Tenant`
- `SaaSUser`
- `Resource`
- `ResourceProfile`
- `Availability`
- `ServicePreset`
- `Booking`

補足説明では、必要に応じて次の存在を口頭で補えば十分です。

- 招待機能
- SSO認証コード
- メールテンプレート
- 通報機能

## 4. ER図の説明ポイント

説明は次の順序が自然です。

1. `Tenant` を中心に、店舗単位でデータが分かれる
2. `SaaSUser` はログイン権限の主体
3. `Resource` は実際に予約されるスタッフ実体
4. `ResourceProfile` で公開プロフィールを管理
5. `Availability` でシフトを管理
6. `ServicePreset` でサービス内容を管理
7. `Booking` で予約情報を管理

## 5. 構成図の見せ方

構成図は、要素数を絞って 3 層で見せると最も読みやすくなります。

1. 利用者・フロント層
2. `Rosterly` の中核機能層
3. 基盤・外部サービス層

今回の構成図に含める要素：

- 顧客（外部サービス利用）
- 顧客（Rosterly 直接利用）
- 店舗スタッフ / 管理者
- `外部サービス連携`
- `公開予約ページ`
- `Rosterly`
- SSO認証
- 予約管理
- 店舗運営管理
- PostgreSQL
- Redis / Celery
- Discord OAuth
- Stripe
- メール / Webhook

## 6. 構成図の説明ポイント

強調したい価値は次の 4 点です。

1. `Rosterly` は単一案件専用ではなく、再利用可能なSaaS基盤
2. 外部サービス経由の顧客は、SSO認証を通じて `Rosterly` の予約機能を利用できる
3. `Rosterly` 直接利用の顧客は、ログイン不要の公開予約ページから予約できる
4. 店舗運営、予約受付、通知連携がひとつの基盤で完結している
