## 2. 总体架构

当前代码库包含两层系统：
- **System B / Rosterly（`system_b_saas`）**：主体 SaaS 核心层，负责租户、排班、预约、订单、通知与后台能力
- **System A / Veludo（`system_a_veludo`）**：基于 Rosterly 的品牌化定制前台与业务编排层

### 2.1 调用链

1. Rosterly 承载租户级业务规则与核心 API。
2. System A 在定制场景下通过 `utils/saas_client.py` 以 HTTP 调用 Rosterly。
3. Rosterly 执行业务校验（租户鉴权、冲突检测、状态流转）后返回 JSON。
4. 异步通知（邮件等）由 Celery Worker 执行。

### 2.2 服务拓扑（默认 `docker-compose.yml`）

- `system_a`：Django/Gunicorn（端口 `8000`）
- `system-b`：Django/DRF/Gunicorn（端口 `8001`）
- `worker`：Celery Worker（消费 System B 任务）
- `db`：PostgreSQL 15
- `redis`：Redis 7

## 3. 代码结构

- `system_a_veludo/`: 用户站点、认证、前端页面、System B 代理调用
- `system_b_saas/`: 核心 API、租户权限、排班与预约模型、后台面板
- `docker-compose.yml`: 一机部署（A/B/DB/Redis/Worker）
- `compose.veludo.yml`: 仅 System A（依赖远端 DB/Redis）
- `compose.rosterly.yml`: 仅 System B 套件（DB/Redis/API/Worker）

## 4. 核心数据模型

### 4.1 System A

- `core.User`
  - 关键字段：`is_cast`, `vrc_id`, `discord_id`, `saas_user_id`, `saas_tenant_id`, `saas_role`, `avatar`
  - 身份映射建议：`saas_user_id` 为主映射键，`discord_id` 仅兼容兜底
- `casts.CastProfile`
  - 关键字段：`saas_resource_id`（映射 System B Resource UUID/外部ID）
  - 预约时长能力：`allow_30_min`, `allow_60_min`, `allow_120_min`

### 4.2 System B

- `tenants.Tenant`
  - 存放租户与 API Key（`X-Tenant-Key`）
  - API 治理字段（2026-04-03）：
    - `is_api_enabled`（店铺 API 开关）
    - `api_ban_reason` / `api_ban_note` / `api_ban_media`
    - `api_banned_at` / `api_banned_by`
- Discord 身份主键口径
  - 稳定唯一键来源：`allauth.socialaccount.SocialAccount.uid`（Discord 长数字 UID）
  - `SaaSUser.discord_id` 在历史数据中可能是展示名，仅作兼容，不建议作为长期唯一映射键
- `resources.Resource`
  - 关键字段：`external_id`（System A 用户ID映射）、`name`, `email`, `linked_user`
  - 说明：排班/预约归属在 `Resource` 层，`linked_user` 用于将登录账号映射到可操作的业务实体
- `resources.ResourceProfile`
  - Cast 公开资料主表（`intro`, `tags`, `avatar_url`, `youtube_url`, `display_order`, `allow_30/60/120`）
- `resources.ResourceMedia`
  - Cast 公开媒体表（`IMAGE/VIDEO`, `image_url`, `video_url`, `cover_url`, `order`）
- `resources.Availability`
  - 排班窗口，支持 `is_recurring`
- `resources.RecurringPattern`
  - 周期规则（`day_of_week`, `valid_from`, `valid_until`）
- `resources.ScheduleTemplate`
  - 可复用周排班模板（`week_config` JSON）
- `resources.ServicePreset`
  - 店铺级服务预设（名称、说明、价格、时长、启用状态、排序）
- `bookings.Booking`
  - 关键字段：`status`, `start_time`, `end_time`, `customer_*`, `selected_service`, `selected_service_name`
  - 通报聚合字段：`customer_report_count`, `cast_report_count`, `last_reported_at`
- `bookings.BookingReport`
  - 用途：承接顾客与 Cast 的双向通报记录（后台消息提醒来源）
  - 关键字段：`reporter_role(CUSTOMER/CAST)`, `reason`, `detail`, `media`, `is_read_by_admin`, `created_at`

> 迁移方向：System B 作为 Cast 主数据源；System A 逐步从“本地 CastProfile + 同步”迁移到“直接读取 `/api/v1/integration/resources/`”。

> 注意：模型里有 `CANCELLED`、`PENDING` 旧状态定义，但当前主流程主要使用 `CONFIRMED` 与 `COMPLETED`（由 PATCH 扩展）。

### 4.3 SaaSUser 与 Resource 绑定说明（2026-03-25）

- `SaaSUser` 是身份账号（认证与权限主体），并不天然等于“可被预约/可排班对象”。
- `Resource` 是业务对象（排班、预约、展示资料、模板等都挂在该对象上）。
- 绑定的必要性：
  - 同一租户可能存在不出勤账号（管理员、运营、仅后台账号），不应默认可排班。
  - 业务资料（`ResourceProfile/Media`）与预约关系以 `Resource` 为中心，和账号解耦更稳定。
  - 跨系统（A/B）同步时，`Resource` 还能承接来自 A 的 `external_id` 映射。
- 当前员工端行为：
  - `STAFF` 账号在注册/登录阶段自动执行绑定：优先复用同租户可匹配资源，否则创建新 `Resource` 并关联 `linked_user`。
  - 自动绑定后，员工可直接编辑自己绑定 `Resource` 的扩展资料并管理自己的排班/预约。
  - 非 `STAFF` 账号（如纯后台管理账号）不强制绑定业务 `Resource`。
  - 绑定主键优先级：`saas_user_id(user.id) -> discord_uid -> discord_id`，用于兼容历史数据并避免显示名漂移。

### 4.4 A/B 服务映射补充（2026-03-26）

- Cast 资料同步：
  - 当 A 侧仅同步 `allow_30/60/120_min` 时，B 侧自动映射同租户 `ServicePreset`（按时长匹配）并回填 `profile.metadata.service_preset_ids`。
  - cast 用户在 A 端个人资料页保存后，会触发一次强制同步（事务提交后执行），避免“保存成功但 B 侧未即时更新”。
- 订单同步：
  - 当未传 `service_id/service_name` 时，B 侧按 `course_duration_minutes`（或 `end-start`）匹配服务预设并补齐 `selected_service_name`。
  - 历史短名（`30分/60分/120分`）在展示层自动补全为 `XX分VRASMR施術コース (PCVR)`，统一运营口径。
- 角色与租户：
  - `CONSUMER` 账号强制 `tenant=null`。
  - `STAFF/ADMIN` 才允许租户绑定与后台权限。

邮箱同步补充（2026-04-03）：
- A/B 身份同步新增邮箱字段协同：A 在回拉 `identity` 时同步远端邮箱。
- 初始值口径：当 B 侧邮箱为空且 A 侧已有邮箱时，由 A 推送至 B（A-first bootstrap）。
- B 侧租户员工账号邮箱更新时，会联动同步其 `linked Resource.email`，避免通知收件人与后台账号邮箱不一致。

### 4.5 店铺设置与邀请模型补充（2026-03-26）

- `Tenant` 新增字段：
  - `contact_email`
  - `logo`
- 新增模型：`tenants.StaffInvite`
  - 用于管理员生成“邀请员工/管理员”链接（含过期时间与使用次数限制）。
  - 受邀者通过 Discord 登录后自动绑定到邀请对应租户与角色。
- 初次店铺注册改造：
  - 点击“注册店铺”后先走 Discord OAuth；
  - OAuth 成功后进入店铺表单（店铺名、logo、owner email、预设服务）；
  - 提交后一次性写入 Tenant + 预设服务 + 管理员身份；
  - 若中途取消，清理本次临时 OAuth 注册记录（provisional user）。

## 5. 关键业务规则

### 5.1 租户鉴权

System B API 默认使用 `IsTenantAuthorized`：
- Header: `X-Tenant-Key`（兼容 `X-API-KEY`）
- 校验成功后将 `request.tenant` 注入后续查询
- 若租户 `is_api_enabled=False`，鉴权直接拒绝（视为 API 封禁态）

### 5.2 预约冲突规则（System B）

创建预约 `POST /api/v1/integration/bookings/` 时：
- 时间冲突判断包含前后 **30 分钟缓冲区**
- 冲突状态检查 `status='CONFIRMED'`
- 冲突返回 `409`

### 5.3 24 小时规则

- 预约创建（System A `BookingActionAPI`）: 必须 `start >= now + 24h`
- 排班创建（System B Availability）: Cast 新增排班也受 24h 限制
- 可预约搜索（mode=`search`）: 24h 内窗口直接不可约

### 5.4 取消与完结规则

- 取消：`start_time - now < 2h` 时拒绝（400）
- Cast 端禁止取消订单（System A 后端拦截）
- 完结：仅允许 `CONFIRMED -> COMPLETED`

### 5.5 角色权限（System A）

- Guest:
  - 可创建/取消自己的预约（受 24h、2h 限制）
- Cast:
  - 可管理自己的排班与模板
  - 可完结自己的预约
  - 不能取消预约

### 5.6 管理端筛选与排序（2026-03-26）

- `Shifts & Orders` 时间筛选粒度为“天”（开始日/结束日）。
- 管理员订单列表按预约开始时间 `start_time` 倒序展示（非创建时间）。

### 5.7 管理台模块更新（2026-03-27）

- 订阅信息从 `Store Settings` 中拆分为独立 `Subscription` 模块。
- 顶部“ログイン中”左侧提供“サブスクリプション管理”按钮，直接切换到订阅编辑模块。
- 所有管理台提交操作启用“返回当前 tab”策略，避免保存后固定跳回店铺设置。

Core-Time 店铺补充：

- `Tenant.store_type=CORE_TIME` 时，`Shifts & Orders` 显示 Core-Time 注文编辑器。
- Core-Time 编辑字段缩减为：担当者、顧客名(vrcID)、服务预设、开始时间。
- 后端自动以服务时长推导结束时间，并写入 `Booking.booking_type=CORE_TIME`。
- Core-Time 订单状态统一按 `CONFIRMED` 处理（后台编辑不再暴露状态输入）。

### 5.8 公开预约链接增强与反刷保护（2026-03-29）

- 店铺设置新增字段：`Tenant.booking_detail_redirect_url`（可选）
  - 配置后，确认邮件“詳細を見る”按钮优先跳转到该 URL（例如 Veludo 自有主页承接页）。
  - 系统会自动附加 `booking_token` 与 `tenant` 查询参数。
- 顾客订单详情页新增取消期限明文展示：
  - 展示文案（日语）：`キャンセル期限：YYYY年MM月DD日 HH:MM まで`
  - 计算口径：`booking.start_time - tenant.cancellation_window_hours`
- anti-bot（公开预约创建）：
  - 前端 honeypot 字段：`website`（隐藏）
  - 后端限流：同租户按 IP（10 分钟 / 1 小时）+ 指纹（10 分钟）双层限制
  - 超限返回：HTTP `429`

## 9. 可观测性与排错

### 9.0 登录映射专项检查（SSO 改造后必须执行）

- 风险背景：若历史上长期使用 `discord_id`（显示名）作为主键/主映射键，用户改名后会出现“同人新号”的感知。
- 诊断优先级：
  1. 先看 A 的 `saas_user_id` 是否命中 B 的 `user_id`
  2. 再看 B 的 `SocialAccount.uid` 与 A 映射是否一致
  3. 最后才看 `discord_id` 展示名匹配
- 处置原则：
  - 发现重复影子账号时，优先保留有业务数据的账号并回写 `saas_user_id`
  - 未确认前不要直接删用户，先执行只读核对和备份

### 9.1 常见故障检查顺序

1. `docker compose ps` 检查容器状态
2. `docker compose logs -f system_a system-b worker` 看错误栈
3. 检查 `SAAS_API_URL` 与 `X-Tenant-Key` 是否一致
4. 检查 Cast 是否存在 `saas_resource_id`
5. 检查时间时区（统一 `Asia/Tokyo`）

### 9.2 高频问题

- 预约成功但没邮件：通常是 Worker 或 Redis 异常
- Cast 看不到排班：`resource_id` 不匹配或跨用户权限拦截
- 预约被拒绝：24h 规则、30min 缓冲冲突、2h 取消限制
- 预约接口返回 `429`：触发了公开预约 anti-bot 限流，需等待窗口期后再试
- 通报红点不更新：优先检查管理员权限（需 `ADMIN`）、`/dashboard/api/reports/notifications/` 返回码、以及前端是否成功触发 `mark_reports_read`
- 点击“すべて既読”无反应：优先检查浏览器 Cookie 中是否存在 `saas_csrftoken`（或兼容 `csrftoken`），以及 POST `/dashboard/` 是否返回 `200`
- Integration API 无法调用且 key 正确：检查目标租户是否被超管封禁（`is_api_enabled=False`）

### 9.3 本地文件上传失败（容器路径误用）

- 现象：服务器可上传附件，本地开发环境上传失败。
- 根因：`STATIC_ROOT` / `MEDIA_ROOT` 固定写死容器路径（如 `/app/...`），本地进程无写权限或目录不存在。
- 修复口径（2026-04-03）：
  - 配置改为环境感知：容器内默认 `/app/...`，本地进程回落到仓库工作目录路径。
  - 保持 `MEDIA_URL` 不变，仅修正存储落盘路径。
- 建议验证：本地提交一次通报附件与头像上传，确认文件实际落盘到项目目录下 `media/`。

## 10. 发布与变更建议

### 10.3 分离部署上线清单（已整合）

适用：
- Server A 使用 `compose.veludo.yml`
- Server B 使用 `compose.rosterly.yml`

发布顺序：
1. 先发 B：`pull -> up --build -> migrate -> check -> ps`
2. 再发 A：`pull -> up --build -> migrate -> check -> ps`
3. A 低内存机器优先使用：`--no-build` 或仅 `--force-recreate`
