# Rosterly 技術アーキテクチャ詳細
## 商用マルチテナント予約・シフト管理SaaS（対外公開版）

## 0. 概要
**Rosterly** は、仮想空間ビジネス向けの **マルチテナント型予約・シフト管理SaaS** です。  
本システムは、単なる予約UIではなく、B2B商用運用で要求される以下を中核に設計しています。

- テナント境界を破らないデータ分離
- ロールごとの厳格な認可制御
- 外部ID基盤を活用した安全な認証
- 単独開発でも再現可能なインフラ運用

技術スタック: Python (Django), HTML/CSS/JavaScript, Nginx, Docker, Oracle Cloud, Discord OAuth

---

## 1. システムアーキテクチャの全体像

### 1.1 論理構成
システム全体は次の責務分離で構成しています。

`Frontend (HTML/CSS/JS) -> Nginx (Reverse Proxy) -> Django Backend -> RDB`

- Frontend: 予約・シフト・管理画面の業務UI
- Nginx: TLS終端、静的配信、リクエスト制御、上流保護
- Django: 認証、認可、予約整合性、テナント境界管理
- RDB: 業務データの永続化と整合性保証

### 1.2 単独開発でこの構成を選んだ理由
1. 開発速度と実装密度
Djangoの標準機能（ORM、認証、管理機能）を活用し、短期間で業務機能を安定実装できるため。

2. 将来拡張に耐える責務分離
初期はモノリスで迅速に立ち上げつつ、負荷増加時に非同期処理・API境界分離へ段階移行しやすいため。

3. 運用面の再現性
Nginx + Dockerにより、ローカル/検証/本番の差分を最小化し、単独体制でも品質を維持しやすいため。

4. B2B要件との整合
B2B SaaSで重視される「セキュリティ」「可監査性」「権限事故の予防」をアプリ層とインフラ層で両立しやすいため。

## 2. B2B要件の中核: マルチテナントとRBACの設計

### 2.1 マルチテナント分離戦略
本システムは、店舗（Tenant）を境界とした論理分離を前提にしています。

1. データモデルでの境界固定
予約・シフト・リソース・運用設定を含む主要テーブルに `tenant_id` を保持し、業務データをテナント単位で管理。

2. アプリケーション層での強制スコープ
一覧・検索・更新・削除すべてで「現在のテナント文脈」を必須化し、テナント条件漏れを構造的に抑止。

3. 組織外ユーザーの明示区分
外部フロント連携ユーザー向けに `CONSUMER` ロールを分離し、従業員権限（`STAFF`/`ADMIN`）と混線しない設計に統一。

4. ライフサイクル制御
テナントのソフト削除時は公開予約導線を停止し、削除申請中データの不用意な外部露出を防止。

### 2.2 RBAC（Role-Based Access Control）
ロールは業務責務に対応させ、最小権限原則で制御しています。

- `ADMIN`:
  - テナント内ユーザー/ロール管理
  - 全体のシフト・予約・サービス設定・テンプレート運用
- `STAFF`:
  - 自身のシフト管理
  - 自身担当予約の参照・更新（範囲限定）
- `CONSUMER`:
  - 予約利用主体としての限定機能
  - 従業員管理機能へのアクセス不可

### 2.3 API越権防止
1. 認証済みであることと、操作可能であることを分離判定
2. オブジェクト単位で `tenant_id` と担当者整合性を検証
3. 判定不能時は拒否する Fail Closed を徹底
4. 越権パターン（他テナント参照、他人ID指定）を回帰テスト対象として固定化

## 3. 認証とセキュリティ基盤

### 3.1 Discord OAuth によるSSO
認証はDiscord OAuthを中核に据え、パスワード自前管理を最小化しています。

- 採用理由:
  - 認証情報漏えいリスクの低減
  - 既存ID基盤活用によるオンボーディング簡素化
  - 認証運用コストの削減

- 実装要点:
  - 認可コードのワンタイム消費
  - 状態値検証によるCSRF対策
  - 識別子マッピング時の優先キー設計（表示名依存を回避）

### 3.2 商用向け防御機構
1. レート制御（Rate Limit）
ログイン・予約作成・コード交換など高リスク経路に対し、IP単位の閾値制御を実施。

2. 予約導線のAnti-Bot
短期/中期のIP制限、送信指紋ベース制御、honeypot導入で自動化攻撃を抑止。

3. API認証の強化
テナントキーに加え、タイムスタンプとHMAC署名を併用し、改ざん・リプレイ耐性を向上。

4. 監査性
認証イベント、予約状態遷移、権限拒否を監査可能な形式で記録し、障害解析と説明責任に対応。

## 4. インフラ運用とコンテナ化

### 4.1 Dockerによる環境同一化
Dockerを採用し、開発・検証・本番の実行差分を極小化しています。

- 依存バージョン固定（Python/ライブラリ/OSパッケージ）
- サービス起動順序の明示
- 環境変数による設定外出し

これにより「ローカルでは再現しない本番障害」を抑制し、デプロイ信頼性を高めています。

### 4.2 Oracle Cloud運用での現実解
単独開発体制では、機能開発より先に「運用不能化」を避ける設計が重要です。

1. デプロイ手順のスクリプト化
2. 検証手順とヘルスチェックの標準化
3. 障害時の切り戻し手順を先に整備
4. ログ観測点を最小構成で固定

結果として、運用工数を増やさずに継続改善可能な基盤を維持しています。

## 5. 直面した技術的課題と解決策

### 5.1 課題A: 同時予約による競合（ダブルブッキング）
**問題**  
同一スタッフ・同一時間帯に対する同時予約で、アプリ層の事前チェックだけでは競合を完全に防げない。

**対応**
1. 予約確定処理をトランザクション化
2. 対象スロットの排他制御と整合性制約を併用
3. 競合時は明示エラーで返却し、再選択導線へ誘導

**効果**  
レースコンディション起因の超過予約を実運用レベルで抑制。

### 5.2 課題B: テナント条件漏れリスク
**問題**  
マルチテナント開発では、1箇所のクエリ条件漏れが情報漏えいにつながる。

**対応**
1. テナントスコープを共通化し、個別実装を削減
2. ロール/テナント不整合リクエストの拒否を標準化
3. 越権ケースを自動テストに組み込み、回帰を防止

**効果**  
「開発者の注意力」ではなく「構造」で境界を担保する運用に移行。

### 5.3 課題C: クロスシステムID同期の一貫性
**問題**  
表示名ベースの曖昧な照合は、将来的な誤マッピングの温床になる。

**対応**
1. 安定識別子を優先したマッピング順序を定義
2. 互換キーは段階的フォールバック用途に限定
3. 同期失敗時の再実行を前提に冪等更新を採用

**効果**  
運用中のデータドリフトと誤同期インシデントを低減。

---

## まとめ
Rosterlyは、予約機能の提供だけでなく、B2B SaaSとして要求される以下の同時達成を設計目標にしています。

- テナント分離の堅牢化
- RBACと越権防止の徹底
- 外部ID連携による認証安全性向上
- 再現性の高いコンテナ運用

特に、データ境界と権限制御を「運用ルール」ではなく「システム構造」で担保する方針は、セキュリティ・監査性・拡張性を重視するB2B組織に対して再現性の高いアプローチです。
