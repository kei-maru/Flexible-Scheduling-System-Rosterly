# 需求文档：Rosterly（System B 主体）与 System A 定制版

## 1. 产品定位

- 产品主体为 `Rosterly`（System B）。
- `System A` 不是平级系统，而是基于 `Rosterly` 核心能力做出的品牌化定制版前台。
- 业务规则、租户模型、排班/预约/订单/通知能力，原则上以 `Rosterly` 为核心沉淀。
- `System A` 负责品牌前台体验、特定业务编排与定制交互，不反向主导 `Rosterly` 的通用产品设计。

## 2. 产品目标

- 以 `Rosterly` 为主体，设计可复用的多租户排班与预约 SaaS。
- 支持店铺级开通、员工邀请注册、排班管理、订单管理、邮件通知、服务预设与店铺运营。
- 保持与 System A 当前核心业务能力大体一致，但将登录与组织管理流程改造成更标准的 SaaS 结构。

## 3. 系统关系

- `Rosterly / System B`
  - 多租户 SaaS 核心层。
  - 负责：店铺、员工、角色、排班、预约、订单、邮件模板、服务预设、管理后台。
- `System A`
  - `Rosterly` 的特别定制版前台。
  - 面向特定品牌客户，调用 `Rosterly Integration API` 完成预约链路。
- 结论：
  - `Rosterly` 是主产品。
  - `System A` 是基于 `Rosterly` 的定制业务壳层。

## 4. 登录与注册设计

### 4.1 登录方式

- 系统采用 `Discord OAuth` 作为注册与登录的统一入口。
- 不设计传统用户名密码注册作为正式主流程。
- 本地账号登录仅保留开发/测试用途，不作为正式产品方案。
- 最终口径（2026-03-24）：
  - C 端用户在 System A 触发登录，链路为 `A -> B 授权 -> A`。
  - 用户体验侧不暴露 B 的后台概念（无“去 B 登录”心智负担）。
  - 切换账号时，A 发起链路会重新触发认证，不依赖旧会话。
  - 登录闸门（2026-04-01）：用户在 A 点击登录后，先进入 `协议同意页`，确认同意后再跳转 B 的 Discord 授权模块。

### 4.2 店长注册店铺

- 首次使用由 `店长` 发起注册。
- 店长通过 `Discord OAuth` 完成身份认证。
- 注册完成后，系统创建：
  - 一个新的 `Tenant / Shop`
  - 当前店长账号
  - 当前店长的初始 `ADMIN` 权限
- 店长是店铺的首个管理员，可进入 `Rosterly` 管理后台。

### 4.3 店员邀请注册

- 店员不能自由注册加入任意店铺。
- 由店长/管理员在后台生成邀请链接。
- 邀请链接至少包含：
  - 所属店铺标识
  - 目标角色（默认 `STAFF`，也可指定 `ADMIN`）
  - 过期时间
  - 一次性或可控次数使用限制
- 被邀请者点击链接后，通过 `Discord OAuth` 完成注册。
- 注册成功后：
  - 自动归属到对应店铺
  - 自动成为邀请指定角色
  - 若角色为 `STAFF`，系统自动绑定/创建其 `Resource`（可被预约对象）
  - 初次进入对应功能页面

### 4.4 管理员授权其他管理员

- 店长/管理员可以将已有员工提升为管理员。
- 也可以直接生成“管理员邀请链接”，让被邀请者注册后即成为 `ADMIN`。
- 管理员授权属于租户内权限行为，不影响其他租户。

## 5. 角色定义与权限

### 5.0 角色补充（跨系统口径）

- `System B` 新增 `CONSUMER` 角色用于标识“由 A 端登录、且无租户绑定”的用户。
- `CONSUMER` 不属于员工体系，不应进入 B 员工后台能力。
- `STAFF/ADMIN` 仍只用于店铺员工/管理员运营体系。
- 当前角色映射（2026-03-25）：
  - A 管理员（`is_superuser` 或 `is_staff 且非 cast`）→ B `ADMIN`
  - A 员工（`is_cast=True`）→ B `STAFF`
  - A 普通用户 → B `CONSUMER`
- A 普通用户（`CONSUMER`）在 B 侧不绑定租户（`tenant=null`），仅用于 A 端身份透传。

### 5.1 Staff（店员）

- 通过邀请链接注册加入店铺。
- 可管理自己的排班。
- 可查看自己的预约订单。
- 可完结自己的订单。
- 可编辑自己的基础个人资料。
- 不可取消订单。
- 不可查看全店数据。
- 不可编辑其他店员介绍资料。
- 不可编辑邮件模板、服务预设、记账信息。

### 5.2 Admin（店长 / 管理员）

- 店长是首个管理员。
- 管理员可被其他管理员授权。
- 可查看与管理租户内全部排班与订单。
- 可编辑店员介绍资料。
- 可管理员工账号与角色。
- 可授权其他管理员。
- 可使用记账功能。
- 可编辑邮件内容模板。
- 可管理服务预设。

## 6. 核心业务功能

### 6.1 与 System A 基本一致的核心能力

- 排班管理
- 预约管理
- 订单查看与状态管理
- 邮件通知
- 店员资料编辑

### 6.2 System B / Rosterly 作为主体新增或强化的能力

- 店铺创建与租户管理
- 邀请链接注册机制
- 店员/管理员组织关系管理
- 通用服务预设
- 记账功能
- 可复用的后台设计体系

## 7. 页面与模块设计

### 7.1 整体视觉风格

- `Rosterly` 整体设计风格采用当前管理员面板风格。
- 视觉基调参考现有 `tenant_dashboard`：
  - 简洁 SaaS 后台布局
  - 左侧导航 + 右侧内容区
  - 清晰卡片式信息分层
  - 浅色主界面、克制高亮色、偏专业的运营工具气质
- 新增页面应尽量沿用该设计体系，避免和 System A 的品牌前台视觉混淆。

### 7.2 Staff 可使用模块

- 排班管理
  - 管理自己的排班
  - 查看自己的预约占用情况
  - 支持周期排班与模板能力
- 预约订单
  - 查看自己的预约记录
  - 仅可完结自己的订单
- 个人资料
  - 编辑基础资料
  - 编辑与自身绑定 `Resource` 的基础展示信息（非全店统一管理项）
  - 个人前端页面统一要求（2026-03-25）：
    - `/home`、`/profile`、`/schedule`、`/bookings` 使用一致的顶部 Header 与导航结构
    - 顶部持续显示当前店铺名
    - 头像菜单使用点击展开（避免 hover 丢失）

### 7.3 Admin 可使用模块

- 管理员面板整体交互架构与 System A 对齐（运营操作路径尽量一致）
- 左上品牌标识固定为 `Rosterly`
- 取消顶部独立 `Shift 管理` 入口，统一收敛到面板模块导航
- `Users & Roles`（员工账号与角色管理）
- `Cast CMS`（店员展示资料与服务勾选管理）
- `Shifts & Orders`（全店排班与订单联动查看）
- `Service Presets`（店铺级服务预设）
- `Email Templates`（预约确认/取消邮件模板）
- 记账功能（后续迭代保持管理员可见）

### 7.4 公开预约ページ（UI 仕様）

- Cast 一览の表示順は `Cast CMS` の拖拽順に従う（`ResourceProfile.display_order`）。
- 「指名予約」「時間予約」のタブには予約種別を示すアイコンを付与する。
- Cast 详情カードの収起はアニメーション完了後にレイアウト復帰させ、瞬間消失や遅延リフローを避ける。
- 预约确认面板应按三组展示：
  - 担当者（含头像）+ 预约服务
  - 用户输入信息（VRCID/DiscordID/Email 按店铺配置动态显示）
  - 预约完整时间段（开始-结束 + 时长）
- 订单详情页需明确展示日语取消期限：`キャンセル期限：YYYY年MM月DD日 HH:MM まで`。

## 8. 资料编辑边界

### 8.1 店员基础个人资料

- 由店员本人可编辑：
  - 用户名
  - 邮箱
  - 基础头像/基础资料
- 由系统维护、店员前端不可编辑：
  - Discord ID（来源于 OAuth 身份，作为稳定映射标识）

### 8.2 店员介绍资料

- 店员可编辑（仅自己的绑定 Resource）：
  - 自我介绍（`intro`）
  - 标签（`tags`）
  - 头像 URL / YouTube URL
  - 可接单服务项（从店铺 `ServicePreset` 多选勾选）
  - 兼容字段 `allow_30/60/120_min` 由系统根据勾选预设时长自动同步
- 管理员可编辑（全店维度）：
  - 任意店员的展示资料与账号角色
  - 影响店铺统一展示质量的运营字段

## 9. 服务预设设计

- `Rosterly` 需要有店铺级服务预设能力。
- 服务预设由管理员维护。
- 每个服务预设至少包含：
  - 服务名称
  - 服务说明（当前运营口径可使用 `...` 占位）
  - 服务价格（用于后续记账与运营扩展）
  - 时长
  - 启用状态
  - 排序
- System B 的一般用户在预约时应明确选择某个服务。
- 预约订单与邮件通知应显示用户实际选择的服务名称。
- 店员在个人资料页可勾选“可提供的服务预设”（复选框），用于运营展示与能力配置。
- 若是 System A 定制版场景，可继续沿用“由预约开始/结束时间相减推导时长”的方式，不强依赖 System A 自身服务表结构。
- A->B Cast 同步时，若仅传 `allow_30/60/120_min`，B 侧会自动映射到同租户 `ServicePreset`（按时长匹配）并回填到 `profile.metadata.service_preset_ids`。
- A->B 订单同步时，若未明确传 `service_id/service_name`，B 侧会按订单时长匹配 `ServicePreset` 自动补齐服务名。
- 订单展示口径统一：
  - 若历史数据仅有 `30分 / 60分 / 120分` 短名，展示时自动补全为 `XX分VRASMR施術コース (PCVR)`。

## 10. 邮件功能设计

- 邮件能力由 `Rosterly` 统一提供。
- 邮件模板由管理员维护。
- 邮件内容支持动态变量。
- 一般化 `Rosterly` 场景下：
  - 邮件中的服务名优先使用用户实际选择的服务预设名称。
  - 模板变量支持 `{{ selected_service_name }}`（首选）与 `{{ duration_minutes }}`（兼容）。
  - 管理端编辑模板时，服务名输入框应优先预填历史预约/预设中的服务名称。
- System A 定制场景下：
  - 若未传特定服务预设，则可回退为根据预约时长动态生成服务文案。
- 邮件“详情按钮”支持店铺级跳转地址：
  - 店铺可配置 `booking_detail_redirect_url`，将顾客导回自有前台承接页；
  - 系统自动附加 `booking_token` 与 `tenant` 参数；
  - 邮件正文需显示取消截止时间（日语）。

## 10.1 公开预约防刷要求（新增）

- 公开预约提交接口必须具备基础 anti-bot 机制：
  - IP 限流（短窗口 + 长窗口）
  - 提交指纹短时限流（同邮箱/ID/时段重复提交）
  - 前端 honeypot 隐藏字段拦截机器人
- 超过阈值时返回 `429`，并给出可理解的重试提示。

## 11. 记账功能

- `Rosterly` 管理员侧需要显示记账功能。
- 该功能面向店长/管理员，不面向普通店员。
- 当前需求文档先定义为“管理员可见的店铺运营模块”，细节字段与报表结构可在后续专题文档中展开。

## 13. System A 作为定制版的保留原则

- System A 继续保留其品牌前台与特定交互。
- System A 预约场景可以不引入额外数据库迁移，只通过预约起止时间推导时长。
- 但 `Rosterly` 作为主产品，应具备完整的通用服务预设与用户选择服务的业务能力。
- 后续新增通用能力优先落在 `Rosterly`，再评估是否需要反向同步到 System A 定制版。

## 14. 验收方向

- 能由店长通过 Discord 完成店铺创建。
- 能由管理员生成邀请链接并邀请店员/管理员注册。
- 店员注册后能进入自己的工作页面。
- 店员只能操作自己的排班、订单与基础资料。
- 店员端 `Discord ID` 不可手动编辑。
- 店员只能完结自己的订单，不可取消订单。
- 管理员可管理全店排班、订单、员工资料、服务预设、邮件模板与记账模块。
- 整体界面风格统一采用当前管理员面板风格。
- System B 在一般化预约链路中可让用户选择具体服务，并在邮件中显示该服务名称。

## 15. 最新管理台补充（2026-03-27）

- 管理员面板新增独立 `Subscription` 模块，入口位于顶部“ログイン中”左侧的“サブスクリプション管理”按钮。
- `Store Settings` 不再承载订阅字段，仅保留店铺基础配置、预约公开配置与店铺条款配置。
- 管理台“保存后返回当前模块”生效：各模块提交后会回到原 tab，不再固定跳回 `Store Settings`。
- 店铺类型继续保留在 `Store Settings`：
  - `コアタイム制`
  - `自由シフト制`
- `コアタイム制` 下的注文编辑字段收敛为 4 项：
  - 担当者
  - 顧客名（vrcID 另字段）
  - サービス名（来自 Service Presets 下拉）

- Core-Time 注文的结束时间由后端基于所选服务 `duration_minutes` 自动计算，订单统一标记为 `booking_type=CORE_TIME`。
