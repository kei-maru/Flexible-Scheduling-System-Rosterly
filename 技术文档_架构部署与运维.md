## 2. 总体架构

当前代码库包含两层系统：
- **System B / Rosterly（`system_b_saas`）**：主体 SaaS 核心层，负责租户、排班、预约、订单、通知与后台能力
- **System A / Veludo（`system_a_veludo`）**：基于 Rosterly 的品牌化定制前台与业务编排层

### 2.1 调用链

1. Rosterly 承载租户级业务规则与核心 API。
2. System A 在定制场景下通过 `utils/saas_client.py` 以 HTTP 调用 Rosterly。
3. Rosterly 执行业务校验（租户鉴权、冲突检测、状态流转）后返回 JSON。
4. 异步通知（邮件等）由 Celery Worker 执行。

### 2.2 服务拓扑（默认 `docker-compose.yml`）

- `system_a`：Django/Gunicorn（端口 `8000`）
- `system-b`：Django/DRF/Gunicorn（端口 `8001`）
- `worker`：Celery Worker（消费 System B 任务）
- `db`：PostgreSQL 15
- `redis`：Redis 7

## 3. 代码结构

- `system_a_veludo/`: 用户站点、认证、前端页面、System B 代理调用
- `system_b_saas/`: 核心 API、租户权限、排班与预约模型、后台面板
- `docker-compose.yml`: 一机部署（A/B/DB/Redis/Worker）
- `compose.veludo.yml`: 仅 System A（依赖远端 DB/Redis）
- `compose.rosterly.yml`: 仅 System B 套件（DB/Redis/API/Worker）

## 4. 核心数据模型

### 4.1 System A

- `core.User`
  - 关键字段：`is_cast`, `vrc_id`, `discord_id`, `saas_user_id`, `saas_tenant_id`, `saas_role`, `avatar`
  - 身份映射建议：`saas_user_id` 为主映射键，`discord_id` 仅兼容兜底
- `casts.CastProfile`
  - 关键字段：`saas_resource_id`（映射 System B Resource UUID/外部ID）
  - 预约时长能力：`allow_30_min`, `allow_60_min`, `allow_120_min`

### 4.2 System B

- `tenants.Tenant`
  - 存放租户与 API Key（`X-Tenant-Key`）
- Discord 身份主键口径
  - 稳定唯一键来源：`allauth.socialaccount.SocialAccount.uid`（Discord 长数字 UID）
  - `SaaSUser.discord_id` 在历史数据中可能是展示名，仅作兼容，不建议作为长期唯一映射键
- `resources.Resource`
  - 关键字段：`external_id`（System A 用户ID映射）、`name`, `email`
- `resources.ResourceProfile`
  - Cast 公开资料主表（`intro`, `tags`, `avatar_url`, `youtube_url`, `display_order`, `allow_30/60/120`）
- `resources.ResourceMedia`
  - Cast 公开媒体表（`IMAGE/VIDEO`, `image_url`, `video_url`, `cover_url`, `order`）
- `resources.Availability`
  - 排班窗口，支持 `is_recurring`
- `resources.RecurringPattern`
  - 周期规则（`day_of_week`, `valid_from`, `valid_until`）
- `resources.ScheduleTemplate`
  - 可复用周排班模板（`week_config` JSON）
- `resources.ServicePreset`
  - 店铺级服务预设（名称、时长、启用状态、排序）
- `bookings.Booking`
  - 关键字段：`status`, `start_time`, `end_time`, `customer_*`, `selected_service`, `selected_service_name`

> 迁移方向：System B 作为 Cast 主数据源；System A 逐步从“本地 CastProfile + 同步”迁移到“直接读取 `/api/v1/integration/resources/`”。

> 注意：模型里有 `CANCELLED`、`PENDING` 旧状态定义，但当前主流程主要使用 `CONFIRMED` 与 `COMPLETED`（由 PATCH 扩展）。

## 5. 关键业务规则

### 5.1 租户鉴权

System B API 默认使用 `IsTenantAuthorized`：
- Header: `X-Tenant-Key`（兼容 `X-API-KEY`）
- 校验成功后将 `request.tenant` 注入后续查询

### 5.2 预约冲突规则（System B）

创建预约 `POST /api/v1/integration/bookings/` 时：
- 时间冲突判断包含前后 **30 分钟缓冲区**
- 冲突状态检查 `status='CONFIRMED'`
- 冲突返回 `409`

### 5.3 24 小时规则

- 预约创建（System A `BookingActionAPI`）: 必须 `start >= now + 24h`
- 排班创建（System B Availability）: Cast 新增排班也受 24h 限制
- 可预约搜索（mode=`search`）: 24h 内窗口直接不可约

### 5.4 取消与完结规则

- 取消：`start_time - now < 2h` 时拒绝（400）
- Cast 端禁止取消订单（System A 后端拦截）
- 完结：仅允许 `CONFIRMED -> COMPLETED`

### 5.5 角色权限（System A）

- Guest:
  - 可创建/取消自己的预约（受 24h、2h 限制）
- Cast:
  - 可管理自己的排班与模板
  - 可完结自己的预约
  - 不能取消预约

## 6. 本地启动与部署

### 6.1 一机启动（推荐）

```bash
docker compose up -d --build
```

默认访问：
- System A: `http://localhost:8000`
- System B API: `http://localhost:8001/api/v1/integration/...`

### 6.2 初始化（首次）

```bash
# System A
docker compose exec system_a python system_a_veludo/manage.py migrate

# System B
docker compose exec system-b python system_b_saas/manage.py migrate

# 可选：创建管理账号
docker compose exec system_a python system_a_veludo/manage.py createsuperuser
docker compose exec system-b python system_b_saas/manage.py createsuperuser
```

### 6.3 分离部署（两台机）

- 仅 A：使用 `compose.veludo.yml`
- 仅 B：使用 `compose.rosterly.yml`
- A 通过 `SAAS_API_URL` 指向 B 的公开地址（或内网地址）

## 8. 异步任务与通知

System B 预约成功/取消后通过 Celery 触发任务：
- `process_new_booking.delay(booking.id)`
- `send_cancellation_email_task.delay(...)`

依赖：
- Redis 必须可用
- `worker` 容器必须常驻

邮件渲染口径（2026-03-20 更新）：
- System B 一般化场景下，确认邮件优先显示 `selected_service_name`
- 若未选择服务，再回退到邮件模板 `service_name`
- 若仍无服务名，再按 `Booking.end_time - Booking.start_time` 动态计算
- 支持模板变量：`{{ duration_minutes }}`、`{{ duration_hours }}`
- 对旧模板里固定 `60分` 文案做兼容替换，无需数据库迁移

## 9. 可观测性与排错

### 9.0 登录映射专项检查（SSO 改造后必须执行）

- 风险背景：若历史上长期使用 `discord_id`（显示名）作为主键/主映射键，用户改名后会出现“同人新号”的感知。
- 诊断优先级：
  1. 先看 A 的 `saas_user_id` 是否命中 B 的 `user_id`
  2. 再看 B 的 `SocialAccount.uid` 与 A 映射是否一致
  3. 最后才看 `discord_id` 展示名匹配
- 处置原则：
  - 发现重复影子账号时，优先保留有业务数据的账号并回写 `saas_user_id`
  - 未确认前不要直接删用户，先执行只读核对和备份

### 9.1 常见故障检查顺序

1. `docker compose ps` 检查容器状态
2. `docker compose logs -f system_a system-b worker` 看错误栈
3. 检查 `SAAS_API_URL` 与 `X-Tenant-Key` 是否一致
4. 检查 Cast 是否存在 `saas_resource_id`
5. 检查时间时区（统一 `Asia/Tokyo`）

### 9.2 高频问题

- 预约成功但没邮件：通常是 Worker 或 Redis 异常
- Cast 看不到排班：`resource_id` 不匹配或跨用户权限拦截
- 预约被拒绝：24h 规则、30min 缓冲冲突、2h 取消限制

## 11. 反爬封禁与日志清理（2026-03-17 更新）

### 11.1 自动识别高频 Bot

System A 已加入埋点防刷机制（`/core/api/track/`）：
- 高频 IP 达到阈值后自动加入 `BlockedIP`
- 被封 IP 访问埋点或业务 API 时直接返回 `403`
- 被封 IP 的埋点不再写入 `UserActivity`

关键配置（`system_a_veludo/config/settings.py`）：
- `TRACKING_BOT_WINDOW_SECONDS`（默认 `600`）
- `TRACKING_BOT_THRESHOLD`（默认 `300`）
- `TRACKING_IP_WHITELIST`（默认 `127.0.0.1,::1`）

### 11.2 一键清理埋点命令

新增管理命令：`cleanup_user_activity`

命令位置：
- `system_a_veludo/core/management/commands/cleanup_user_activity.py`

常用场景：

1) 清理某个污染 IP（并封禁）

```bash
docker compose exec system_a python system_a_veludo/manage.py cleanup_user_activity --ip 38.190.254.158 --ban-ip
```

2) 仅预览（不删除）

```bash
docker compose exec system_a python system_a_veludo/manage.py cleanup_user_activity --ip 38.190.254.158 --dry-run
```

3) 清理最近 N 小时某 IP 数据

```bash
docker compose exec system_a python system_a_veludo/manage.py cleanup_user_activity --ip 38.190.254.158 --hours 24 --ban-ip
```

4) 自动识别并清理高频 Bot（按窗口）

```bash
docker compose exec system_a python system_a_veludo/manage.py cleanup_user_activity --auto-bot --window-minutes 60 --min-count 200
```

5) 清空所有埋点（危险操作）

```bash
docker compose exec system_a python system_a_veludo/manage.py cleanup_user_activity --all
```

### 11.3 服务端运行步骤（推荐）

```bash
cd /home/ubuntu/Veludo_Saas
docker compose up -d system_a
docker compose exec system_a python system_a_veludo/manage.py cleanup_user_activity --ip <目标IP> --ban-ip
```

执行后可验证：

```bash
docker compose exec system_a python system_a_veludo/manage.py shell -c "from core.models import UserActivity; print(UserActivity.objects.filter(meta_data__ip='<目标IP>').count())"
docker compose exec system_a python system_a_veludo/manage.py shell -c "from core.models import BlockedIP; print(BlockedIP.objects.filter(ip='<目标IP>', is_active=True).exists())"
```


## 12. Dashboard 与共同排班权限（2026-03-01 更新）

### 11.1 共同排班页面（员工 + 管理员）
- 入口：`/schedule/`
- 用途：统一排班页面，交互与 System A `schedule` 保持一致（拖拽新增、点击删除、周期排班、模板读写）。
- 访问：`STAFF` 与 `ADMIN` 均可访问。
- 数据规则：
  - `STAFF` 仅能管理自己的资源排班。
  - `ADMIN` 可切换并管理租户内所有资源排班。

### 11.2 管理员 Dashboard（仅管理员）
- 入口：`/dashboard/`
- 访问：仅 `ADMIN`（或 `is_superuser`）可访问。
- `STAFF` 访问 `/dashboard/` 时会重定向到 `/schedule/`。
- Dashboard 功能：
  - 员工资料编辑（用户名、邮箱、Discord、角色、启用状态、资源绑定）
  - 订单列表查看
  - 即将排班查看
  - 服务预设管理（名称、时长、启用状态）
  - 邮件模板编辑

### 11.2.1 System B 服务预设与邮件服务名（2026-03-20 追加）
- 新增服务预设模型：租户可维护多个预约服务（如 30/60/90 分钟）
- 预约创建支持 `service_id`（推荐）/`service_name`（兼容）入参
- 订单落库 `selected_service_name`，用于记录用户实际选择的服务
- 预约确认邮件渲染优先级：
  - 优先：`selected_service_name`（用户选择）
  - 回退：邮件模板 `service_name`（支持变量）
  - 再回退：按 `start_time/end_time` 动态时长文案

### 11.3 System A 利用規約（2026-03-15 追加）

### 11.5 管理员数据分析面板（2026-03-16 更新）
- 入口：`/admin-dashboard/` → Analytics 标签页
- 采用模块化展示结构，便于后续追加新指标模块
- 现有模块：网站访问数（来源：`UserActivity`，`action=VIEW_PAGE`）
- 统计项：累计访问、今日访问、近 7 日访问、去重 IP（总/今日/近 7 日）
- 过滤规则：访问统计与平均停留时间默认仅统计交互次数 ≥ 3 的 IP
- 新增指标：手机用户比例（按唯一 IP 去重，不受 ≥ 3 过滤）、平均停留时间（过滤 < 3 秒）
- 埋点新增：`PAGE_DURATION`（页面停留时长），`meta.user_agent` 与 `meta.page`
- 新增图表：访问趋势折线图（最近 14 天，按 `VIEW_PAGE` 统计）
- 新增图表：访问时间分布图（按小时统计，可配置日期范围）

### 11.6 Cast 主数据迁移（2026-03-23 更新）
- 目标：System B（Rosterly）作为 Cast 主数据源，System A 改为 API 读取。
- System A 新增配置：
  - `CAST_SOURCE=remote|local`（默认 `remote`）
  - `CAST_SOURCE_FALLBACK_LOCAL=True|False`（默认 `True`）
- `admin_dashboard` 已改为远端优先读取 Cast，并在远端模式下将拖拽排序写回 System B `display_order`。
- System A 新增迁移命令（幂等，可重复执行）：

```bash
docker compose exec system_a python system_a_veludo/manage.py sync_casts_to_system_b --only-active
```

- 迁移命令能力：
  - 将 `CastProfile/CastMedia` 推送到 System B `/api/v1/integration/resources/`
  - 自动回写 `saas_resource_id`
  - 支持 `--dry-run`、`--limit`
- 管理员编辑 CastProfile 后，System A 会立即同步到 System B（不再依赖仅 `User` 保存信号）。
